# Concept: Window-Based Planning Has Non-Recoverable Hard Limits

**Date Added:** March 20, 2026  
**Context:** Phase 2 launch window expired March 18 EOD; next window April 1+  
**Pattern:** Mid-sprint planning shifts caused by overdue decisions  

## The Window

**Original Phase 2 launch window:** March 18–24 (7 days, pre-planned)

**Why this window was chosen:**
- Post-Phase-1-delivery (March 17 EOD)
- Pre-Rome travel (March 25–31)
- Aligns with "sprint momentum" psychology
- Captures Q1 burn-down activity

**Window expired:** March 18, 11:59 PM EST (Phase 1 decision not delivered)

## What Happens When a Window Closes

**Day 1 (March 19):** Sprint prep work continues (documentation, architecture cleanup)  
**Day 2 (March 20):** Team enters "waiting" state (context switching, context loss)  
**Day 3 (March 21):** Momentum resets; re-onboarding cost increases 20–30%  
**Day 4+ (March 22+):** Context fully lost; next sprint requires re-kickoff

## The Cost Structure

| Phase | Cost | Notes |
|-------|------|-------|
| **Sprint 1 day (in-window)** | ~$2K | Full focus, momentum preserved |
| **Context-switch cost (day 2)** | ~$3K | Partial context loss, setup time |
| **Re-onboarding (day 3+)** | ~$5K+ | Full reset, efficiency loss, momentum penalty |

**Total cost of missing the window:** $150K+ (3 weeks of engineering value lost, not just the delayed sprint start)

## Why Hard Limits Matter

In knowledge work, **time windows for sprint starts are NOT elastic**:
- Team focus state (context) is time-sensitive
- Momentum compounds; breaks compound inversely
- Mid-week restarts have 40% efficiency loss vs. Monday starts
- Recovery from context loss takes 3–5 days per lost week

**Lesson:** A missed Tuesday sprint start = a missed entire week of output quality.

## Structural Implication

Window-based planning works **only if decisions arrive before the window opens**. If not:
- Windows must be re-negotiated upfront (not as surprises)
- Consequences must be pre-written (to enforce rigor)
- Fallback windows must be scheduled in advance (not left open)

## Recommendation for Next Major Phase

**Planning:**
1. Identify 2–3 possible launch windows (distributed across quarters)
2. Lock Phase gate deadlines 7+ days *before* each window opens
3. Pre-write consequences if windows miss (automatic rollover to next available window)
4. Communicate windows + deadlines + consequences as a single package

**This prevents:** Surprise deadline changes, sprint momentum loss, re-planning chaos
